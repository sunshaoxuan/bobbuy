# GAP-02: 客户侧全链路流程差异分析

**生效日期**: 2026-04-19
**状态**: 草案

---

## 1. 概述

本分析旨在对比 BOBBuy V9.0 的当前实现与用户描述的“客户侧真实使用过程”。通过识别功能、架构及交互层面的差异，为 V10.0 的开发提供技术依据。

## 2. 核心差异分析

### 2.1 商品发现与推荐 (Product Discovery)
- **描述**: 客户打开 APP 后应看到按“推荐 -> 最新更新 -> 其他”排序的商品流。
- **现状**: `Product` 模型缺乏 `isRecommended` 字段。当前排序逻辑为默认全量列表。
- **缺失**: 排序字段、权重计算、自动更新时间戳 (`updatedAt`) 的展示逻辑。

### 2.2 预订单与行程解耦 (Pre-Order Flow)
- **描述**: 点击购买后，若无合适行程，客户可输入“希望收货期间”形成预订单。
- **现状**: `OrderHeader` 强制关联 `tripId`。购买流程默认要求选择行程。
- **缺失**: `OrderHeader` 中 `tripId` 的可选性支持；新增 `desiredDeliveryWindow` 字段；前端“收货意向”选择界面。

### 2.3 实时交互与临时商品 (Interaction & Temp Items)
- **描述**: 客户可在聊天中提出列表外商品，采买人员通过新拍照片确认加单。
- **现状**: 当前已具备 ChatMessage 持久化、Trip chat 图片确认与临时商品发布闭环，但同步方式仍是 REST + 轮询，不是 WebSocket 秒推。
- **缺失**: 更细粒度的失败恢复、附件治理与更完善的候选审计分析。

### 2.4 AI 动态协同 (AI Orchestration)
- **描述**: 客户修改预购清单时，AI 实时规划总清单给采买人员。
- **现状**: 采买端仅展示已确认订单。缺乏跨订单的实时动态汇总视角。
- **缺失**: 采购看板的“待采汇总”实时计算逻辑；基于买家修改的变动通知推送。

### 2.5 结算锁死与账单透明 (Post-Procurement Governance)
- **描述**: 采买结束后由于涉及现金结算，禁止删除商品。收货后可在 APP 查看最终账单。
- **现状**: `OrderStatus` 缺乏严格的“结算冻结期”。账单查看分布在审计日志中，缺乏用户友好的“最终结算单”视图。
- **缺失**: `OrderLine` 在 `PROCUREMENT_FINISHED` 状态下的不可篡改逻辑；“收货付款”环节的最终账单页面 (Final Bill View)。

---

## 3. 技术风险评估

- **事务一致性**: 实时 AI 规划总清单涉及多用户并发写入时的锁竞争。
- **即时通讯性能**: 当前风险集中在轮询失败恢复、移动端遮挡与重复刷新治理；WebSocket 重连机制仍属后续演进议题。
- **模型演进**: 修改核心订单模型 (`OrderHeader`) 需要考虑既有数据的迁移。

## 4. 最终确定方案 (Finalized Decisions)

根据用户反馈，V10.0 将按以下逻辑执行：

- **临时商品归属**: 采用“先私有后公开”策略。临时商品仅在聊天双方可见，直到采买人员完善信息并点击“发布到商场”后，转为正式商品公开。支持发布前的 AI 重复项合并预审。
- **AI 总表与穿透逻辑**: 引入“采买批次”概念。由采买人员从预定池中筛选订单进入行程。总清单展示汇总数量，并支持“穿透查询”单人购买明细。
- **结算逻辑**: 简化流程。由采买人员在后台手动确认实付金额与支付状态，无需集成复杂的三方支付网关。
