# STD-01: 开发质量标准

**生效日期**: 2026-01-13
**状态**: 执行中

---

## 1. 概述

本文档定义了项目的开发质量标准。"完成" (Done) 不意味着代码编写完毕，还必须在易用性、美观度和可靠性上达到高标准。

### 1.1 用户画像与设计哲学
- **目标用户**: 非技术背景用户，不熟悉复杂业务系统。
- **设计哲学**: **极简主义 (Minimalism)** 与 **零学习成本 (Zero Learning Curve)**。
- **核心要求**: 页面必须极致简洁，严禁堆砌复杂表单或冗余信息。操作路径应直观化，尽可能通过单键或自动化流转完成任务。

## 2. 评估维度

### 2.1 功能性 (Functionality) - "做什么"
- **完整性**: 用户故事/任务中的所有需求均已实现。
- **正确性**: 功能在正常路径和边缘情况下均按预期工作。
- **集成性**: 功能在整个系统中能正确交互。

### 2.2 易用性 (Usability) - "怎么用"
- **可配置性**: 用户必须能够控制功能行为（例如：按钮动作必须是可配置的）。
- **反馈**: 系统对所有操作提供清晰的反馈（成功、错误、加载状态）。
- **效率**: 常用任务应能以最少的点击完成。
- **状态感知**: 组件应适应其上下文（例如：在详情视图中只读，在编辑视图中可编辑）。

### 2.3 美观性 (Aesthetics) - "长啥样"
- **视觉打磨**: 符合标准交互设计。正确使用标准 UI 组件库。
- **一致性**: 边距、填充、字体和颜色必须符合设计系统。
- **响应式**: 布局必须适应不同的屏幕尺寸。
- **"高级"感**: 避免使用原生交互反馈。使用打磨后的丰富 UI 组件。

### 2.4 可靠性 (Reliability) - "可信度"
- **错误处理**: 无静默失败。错误应被捕获并以友好的方式显示。
- **数据完整性**: 属性必须被正确分类和保护。
- **国际化 (I18n)**: 无硬编码字符串。所有文本必须可翻译。

## 3. 通用功能标准

### 3.1 界面配置
- **控件**: 每个可配置项必须有完整的属性编辑器。
- **预览**: 配置态预览必须与运行时渲染高度一致。
- **动作**: 交互式元素必须具有可配置的触发动作。

### 1.2 后端质量红线 (铁律)
- **行覆盖率 (Line Coverage)**: 必须 ≥ **90%**。
- **分支覆盖率 (Branch Coverage)**: 必须 ≥ **70%**。
- **并发安全性**: 涉及共享资源修改的方法必须有并发单测验证。
- **审计完备性**: 关键业务状态流转必须触发 Audit Log 记录。
- **继承**: 基础配置与扩展配置之间应有清晰的区分和继承逻辑。

### 3.3 数据模型定义
- **分类**: 核心属性与扩展属性之间应有清晰的视觉区分。
- **元数据**: 准确反映底层元数据描述（来源、类型等）。

## 4. "完成"检查清单
- [ ] 功能需求已满足？
- [ ] 易用性已验证（用户真的能用吗）？
- [ ] 美观性已检查（看起来好吗）？
- [ ] 可靠性已测试（错误处理、边缘情况）？
- [ ] 国际化已完成（无硬编码）？
- [ ] **测试覆盖率已计算并体现在评审中**（强制：评审时必须展示 JaCoCo/Vitest 实测数值）？
