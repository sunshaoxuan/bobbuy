# STD-05: 数据库设计规范

**版本**: 1.0
**生效日期**: 2026-01-13
**状态**: 执行中

---

## 1. 物理结构规范

- **命名称规范**: 表名及字段名建议使用全小写及下划线分隔符（snake_case）。
- **主键策略**: 统一使用全局唯一标识符（如 UUID/GUID）或自增序列作为物理主键。
- **关联引用**: 物理外键可选，但逻辑关联必须清晰记录在文档中。

## 2. 基础审计字段

每张业务表必须包含以下标准化审计字段：

| 字段名 | 描述 |
|--------|------|
| `id` | 主键标识 |
| `created_at` | 创建时间戳 |
| `updated_at` | 最后修改时间戳 |
| `created_by` | 创建人标识 |
| `updated_by` | 修改人标识 |
| `is_deleted` | 逻辑删除标志位 (0: 正常, 1: 已删除) |

## 3. 动态扩展原则 (Enum/Metadata)

- **逻辑枚举**: 避免为简单的状态值创建独立的物理表，应使用统一的字典服务或逻辑枚举管理。
- **元数据存储**: 对于高动态性的属性，建议使用扩展字段（如 JSON/XML 容器）或独立的元数据表存储。

## 4. 索引设计

- ✅ **覆盖索引**: 为高频查询的谓词字段（WHERE 子句涉及的字段）建立索引。
- ✅ **唯一约束**: 业务上的唯一性（如账号名、编码）必须由数据库层面的唯一索引保障。

---
**数据库设计应遵循“逻辑优先，物理适配”的原则。**
