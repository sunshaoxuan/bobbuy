# 聊天商品闭环交付说明

## 当前能力
- 聊天链路采用 REST 持久化消息 + 轮询刷新 + 页面事件补偿。
- Trip chat 支持图片识别、候选人工确认、临时商品创建、手动发布到商城。
- 图片确认消息会写入统一审计字段：操作人、触发入口、候选理由、命中片段、人工驳回数量、最终发布时间。
- `itemNumber` 仍是唯一稳定自动命中规则；名称级查重只做候选提示，不做静默自动合并。

## 人工确认点
- 图片识别后先进入“待确认”状态。
- 若出现名称级候选，界面展示推荐原因，必须由采买人员人工选择或放弃。
- 若未直接发布，商品保持 `DRAFTER_ONLY`，后续由聊天卡片手动发布到商城。

## 当前限制
- 暂未引入 WebSocket、消息队列或向量检索。
- 发布失败时保留上下文与消息卡片，并在原卡片上提供明确的重试恢复动作。
- 轮询失败时保留最近一次成功快照，不承诺秒级推送。

## 后续里程碑

- 在更大规模前再评估 WebSocket / 缓存层，而不是提前宣称已交付。
